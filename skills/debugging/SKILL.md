# Rails Debugging Tools & Techniques

This skill provides Rails-specific debugging **techniques and tools**. It complements a systematic debugging methodology by supplying the concrete commands, gems, and patterns you reach for at each stage of investigation.

## When to use this skill

- Rails application behaving unexpectedly
- Tests failing with unclear errors
- Performance issues or slow requests
- N+1 queries suspected or detected
- Production errors need investigation
- Need to inspect SQL generated by ActiveRecord
- Middleware or request pipeline issues
- Asset pipeline or import map problems

## Principles

1. **Read the logs first** — 80% of Rails bugs are answered by `log/development.log` before you touch a debugger.
2. **Reproduce before investigating** — get a failing test or repeatable console session before hypothesizing.
3. **Check the SQL** — when data looks wrong, inspect the actual query with `to_sql` or `ActiveRecord::Base.logger` before blaming application code.
4. **Use the narrowest tool** — `puts` debugging is fine for quick checks; reach for `debug`/`binding.pry` only when you need to step through control flow.
5. **Automate detection** — add `bullet` and `rack-mini-profiler` to development so problems surface before you go looking for them.
6. **Clean up after yourself** — remove all breakpoints, debug logging, and `binding.pry` calls before committing.

## Log Analysis

### Reading Rails logs

```bash
# Follow development logs in real-time
tail -f log/development.log

# Filter by severity
grep -E "ERROR|FATAL" log/production.log

# Filter by request path
grep "Started GET \"/users" log/development.log

# Filter by request ID (useful for concurrent requests)
grep "req_abc123" log/production.log

# Production logs via Kamal
kamal app logs --tail
```

### Adjusting log level at runtime

```ruby
# In console — temporarily increase verbosity
Rails.logger.level = :debug

# In config/environments/development.rb
config.log_level = :debug
config.log_tags = [:request_id]
```

## Console Debugging

The Rails console is your fastest feedback loop for data and query issues.

```ruby
# Start console
bin/rails console

# Test a model
user = User.find(1)
user.valid?                    # Check validations
user.errors.full_messages      # See validation errors
user.changes                   # See unsaved attribute changes

# Inspect associations
user.posts.loaded?             # Has this been eager-loaded?
user.posts.to_sql              # See SQL without executing

# Reload to clear cached state
user.reload

# Sandbox mode — rolls back all changes on exit
bin/rails console --sandbox
```

## SQL Inspection

### Viewing generated SQL

```ruby
# See SQL for any relation without executing
User.where(active: true).order(:name).to_sql
# => "SELECT \"users\".* FROM \"users\" WHERE \"users\".\"active\" = TRUE ORDER BY \"users\".\"name\" ASC"

# Enable live SQL logging in console
ActiveRecord::Base.logger = Logger.new(STDOUT)

# Now every query prints its SQL
User.where(active: true).count
# => SELECT COUNT(*) FROM "users" WHERE "users"."active" = TRUE
```

### Query plan analysis

```ruby
# EXPLAIN a query to see the database execution plan
User.where(active: true).explain
# Shows index usage, seq scans, join strategies

# For complex queries, check for missing indexes
User.where(email: "test@example.com").explain(:analyze)
```

## Breakpoint Debugging

### Using the `debug` gem (Rails 7+ default)

```ruby
# Add a breakpoint anywhere in Rails code
def create
  @user = User.new(user_params)
  debugger                        # Execution pauses here
  @user.save
end

# Commands at the debugger prompt:
# n          — next line (step over)
# s          — step into method call
# c          — continue execution
# p variable — print value
# pp variable — pretty-print value
# info       — show local variables
# bt         — show backtrace
# q          — quit debugger
```

### Using `binding.pry` (via pry-byebug)

```ruby
# Gemfile
gem "pry-byebug", group: [:development, :test]

# In code
def process_order(order)
  binding.pry                     # Drops into Pry REPL
  order.calculate_total
end

# Pry commands:
# next, step, continue (same as debug gem)
# ls           — list methods on current object
# cd object    — change context into object
# show-source  — show method source code
# wtf?         — show last exception backtrace
```

### Using `rails runner` for quick checks

```bash
# Run a one-liner in Rails environment
bin/rails runner "puts User.count"

# Run an investigation script
bin/rails runner scripts/investigate_issue.rb

# With specific environment
RAILS_ENV=production bin/rails runner "puts User.last.email"
```

## N+1 Query Detection

### Manual detection

Watch logs for repeated similar queries:

```
# Bad — N+1 pattern in logs:
User Load (0.3ms)  SELECT "users".* FROM "users"
Post Load (0.1ms)  SELECT "posts".* FROM "posts" WHERE "posts"."user_id" = 1
Post Load (0.1ms)  SELECT "posts".* FROM "posts" WHERE "posts"."user_id" = 2
Post Load (0.1ms)  SELECT "posts".* FROM "posts" WHERE "posts"."user_id" = 3
```

### Fix with eager loading

```ruby
# ❌ N+1: 1 query for users + N queries for posts
@users = User.limit(20)
@users.each { |u| u.posts.size }     # 21 queries

# ✅ Eager load: 2 queries total
@users = User.includes(:posts).limit(20)
@users.each { |u| u.posts.size }     # 2 queries

# Choose the right strategy:
User.includes(:posts)                # Separate queries (default, usually best)
User.eager_load(:posts)              # LEFT OUTER JOIN (filter on association)
User.preload(:posts)                 # Always separate queries (no join filtering)
User.includes(posts: :comments)      # Nested eager loading
```

### Automated detection with Bullet

```ruby
# Gemfile
gem "bullet", group: :development

# config/environments/development.rb
config.after_initialize do
  Bullet.enable = true
  Bullet.alert = true              # Browser popup
  Bullet.bullet_logger = true      # log/bullet.log
  Bullet.rails_logger = true       # Rails log
  Bullet.add_footer = true         # Page footer
end
```

Bullet detects N+1 queries, unused eager loading, and suggests `counter_cache` opportunities automatically.

## Performance Profiling

### Rack Mini Profiler

```ruby
# Gemfile
gem "rack-mini-profiler", group: :development

# Shows timing badge on every page:
# - Total request time
# - SQL query count and time
# - View rendering time
# - Per-query breakdown

# Access detailed profiling at any page by appending:
# ?pp=flamegraph        — Flamegraph visualization
# ?pp=profile-gc        — Garbage collection profiling
# ?pp=analyze-memory    — Memory analysis
```

### Benchmarking in console

```ruby
# Quick benchmark
Benchmark.measure { User.where(active: true).to_a }

# Compare approaches
require "benchmark"
Benchmark.bm do |x|
  x.report("includes") { User.includes(:posts).limit(100).to_a }
  x.report("preload")  { User.preload(:posts).limit(100).to_a }
end
```

## Route & Middleware Debugging

### Routes

```bash
# List all routes
bin/rails routes

# Filter routes by controller or pattern
bin/rails routes -c users
bin/rails routes -g password

# Check where a URL resolves
bin/rails routes | grep "POST.*sessions"
```

### Middleware stack

```bash
# Show the full middleware stack
bin/rails middleware

# Output shows order of Rack middleware:
# use ActionDispatch::HostAuthorization
# use Rack::Sendfile
# use ActionDispatch::Executor
# ...
```

## Migration & Schema Debugging

```bash
# Check migration status — spot pending or rolled-back migrations
bin/rails db:migrate:status

# Show current schema version
bin/rails db:version

# Rollback last migration and retry
bin/rails db:rollback
bin/rails db:migrate

# When schema.rb conflicts after merge
bin/rails db:schema:load     # Destructive — reloads from schema.rb
bin/rails db:migrate          # Preferred — runs pending migrations
```

## Test Debugging

```bash
# Run single test file with verbose output
bin/rails test test/models/user_test.rb --verbose

# Run a specific test by name
bin/rails test test/models/user_test.rb -n test_validates_email

# Run with a fixed seed for reproducing order-dependent failures
bin/rails test --seed 12345

# Run with Ruby warnings to catch deprecations
RUBYOPT=-W bin/rails test
```

```ruby
# In test files — inspect state mid-test
test "creates order" do
  order = Order.create!(params)
  pp order.errors.full_messages    # Quick debug output
  debugger                         # Breakpoint in test
  assert order.persisted?
end
```

## Gotchas

- **Leaving `debugger`/`binding.pry` in committed code** — CI hangs waiting for input. Always search for breakpoints before pushing: `grep -rn "binding\.pry\|debugger" app/ lib/`.
- **Forgetting `ActiveRecord::Base.logger` is global** — setting it to `STDOUT` in console is fine, but doing it in application code floods production logs.
- **Using `puts` in production** — use `Rails.logger.info` instead; `puts` writes to STDOUT which may not be captured.
- **Ignoring Bullet warnings** — N+1 alerts are actionable; don't train yourself to dismiss the popups.
- **Not checking `explain` on slow queries** — a missing index turns a 2ms query into a 2s table scan. Always verify with `EXPLAIN ANALYZE`.
- **Debugging in `production` mode locally** — `RAILS_ENV=production` disables reloading and caching behaves differently. Most debugging belongs in `development`.
